## 1. Зачем менеджеру репозиторий

Репозиторий — это не про код. Это структурированное рабочее пространство, в котором Claude Code:

- **Помнит контекст** — через CLAUDE.md, README.md, FACTS.md он знает твои проекты, метрики, правила
- **Работает по правилам** — не нужно каждый раз объяснять формат ответа, как работать с Excel, куда сохранять файлы
- **Не теряет информацию** — всё сохраняется на GitHub, версионируется, доступно из любого места
- **Масштабируется на команду** — коллеги клонируют репо и получают того же настроенного ассистента

**Без репо:** каждая сессия Claude начинается с нуля. Ты повторяешь контекст, теряешь результаты, файлы разбросаны.

**С репо:** Claude открывает папку и сразу знает: кто ты, какие проекты, какие правила, где бэклог, как отвечать.

## 2. Быстрый старт за 30 минут

### Шаг 1: Создай папку и инициализируй Git

```bash
mkdir my-ecom-repo && cd my-ecom-repo
git init
```

### Шаг 2: Создай минимальный CLAUDE.md

```markdown
# CLAUDE.md

## Repository Overview
Рабочее пространство e-commerce менеджера. Проекты, аналитика, бэклог задач.

## Response Format
1. Unclearness of the task: <0-1>
2. Uncertainty of answer: <0-1>
3. Role: <роль>
4. TL;DR
5. Полный ответ

## Architecture
- *_PRJ/ — папки проектов
- backlog/ — бэклог задач
- scripts/ — утилиты

## File Rules
- Каждая папка должна иметь README.md
- Excel: всегда openpyxl.load_workbook(), никогда Workbook()
- Сохранять Excel с суффиксом _vX.Y.xlsx
```

### Шаг 3: Создай базовую структуру

```bash
mkdir -p backlog scripts _PROMPTS/_ROLES .claude/skills .claude/rules
```

### Шаг 4: Создай .gitignore

```
.env
.DS_Store
__pycache__/
.venv/
*.tmp
node_modules/
```

### Шаг 5: Первый коммит и пуш

```bash
git add -A
git commit -m "init: базовая структура репозитория"
# Создай репозиторий на GitHub (приватный), затем:
git remote add origin git@github.com:USERNAME/my-ecom-repo.git
git push -u origin main
```

Готово. Теперь можно открывать Claude Code в этой папке и он уже видит CLAUDE.md.

## 3. Структура папок

```
my-ecom-repo/
├── CLAUDE.md                    # Главная инструкция для Claude Code
├── README.md                    # Обзор репозитория и навигация
├── .gitignore
├── .claude/
│   ├── skills/                  # Скиллы (переиспользуемые команды)
│   │   ├── ecom-manager/
│   │   ├── project-manager/
│   │   └── excel-worker/
│   ├── rules/                   # Автоматически загружаемые правила
│   │   └── error-learning.md
│   └── memory/                  # Автопамять (между сессиями)
│       └── MEMORY.md
├── _PROMPTS/
│   └── _ROLES/                  # Экспертные роли
│       ├── Head_of_E-Commerce.md
│       ├── Project_Manager.md
│       └── README.md
├── scripts/
│   ├── backup_to_git.zsh
│   ├── convert_xlsx_to_md.py
│   └── README.md
├── backlog/
│   ├── backlog.xlsx             # Единый бэклог задач
│   ├── new-tasks.md             # Входящие задачи
│   └── README.md
├── MARKETPLACE_PRJ/             # Проект: работа с маркетплейсами
│   ├── README.md
│   └── FACTS.md
├── DTC_PRJ/                     # Проект: собственный e-com
│   ├── README.md
│   └── FACTS.md
└── ANALYTICS_PRJ/               # Проект: аналитика и отчёты
    └── README.md
```

**Принципы:**

- Проектные папки именуются `НАЗВАНИЕ_PRJ/`
- В каждой папке — `README.md` (навигация) и при необходимости `FACTS.md` (ключевые факты)
- Точка входа — корневой `README.md` со ссылками на все проекты
- Не плоди папки заранее — создавай по мере появления проектов

## 4. CLAUDE.md — главная инструкция

CLAUDE.md — файл в корне репозитория, который Claude Code читает автоматически при старте каждой сессии. Это «должностная инструкция» для ассистента.

### Обязательные секции

**1. Repository Overview** — кто ты, чем занимаешься, что в репозитории.

**2. Response Format** — как должен выглядеть ответ:
```
1. Unclearness of the task: <0-1> — если >0.3, сначала задать уточняющие вопросы
2. Uncertainty of answer: <0-1>
3. Role: <экспертная роль>
4. TL;DR
5. Полный ответ
```

**3. Architecture** — краткое описание структуры папок и ключевых файлов.

**4. Common Commands** — команды, которые ты часто используешь (бэкап, конвертация, отчёты).

**5. File Editing Rules** — правила работы с файлами:
- Excel: `openpyxl.load_workbook()`, сохранение с `_vX.Y.xlsx`
- Markdown: формат, шаблоны
- Обновлять README.md при изменении структуры

**6. Git Rules** — что Claude Code НЕ должен делать без явной команды: push, commit, rebase, reset.

### Советы

- Держи CLAUDE.md компактным — до 150 строк полезного текста
- Обновляй при изменении структуры или правил
- Не дублируй то, что уже описано в ролях или скиллах — ссылайся

## 5. README.md как Memory Bank

README.md в каждой папке — это «память» Claude Code об этом месте. Когда он открывает папку, первым делом читает README.md и понимает:

- Что здесь находится
- Какие файлы ключевые
- Куда вести по ссылкам дальше

### Шаблон README.md для проектной папки

```markdown
# Название проекта

Last Updated: YYYY-MM-DD

## Описание
Одно предложение — зачем этот проект.

## Структура
- `analytics/` — отчёты и дашборды
- `reports/` — готовые отчёты для руководства
- `FACTS.md` — ключевые факты и цифры

## Ключевые файлы
- [unit-economics.xlsx](analytics/unit-economics.xlsx) — юнит-экономика
- [FACTS.md](FACTS.md) — контакты, реквизиты, ключевые метрики

## Связанные проекты
- [REF: ANALYTICS_PRJ/] — общая аналитика
- [REF: backlog/backlog.xlsx] — задачи по проекту
```

**Правило:** при любом изменении структуры папки — обновляй её README.md.

## 6. FACTS.md — хранение фактов

FACTS.md хранит конкретные факты, которые Claude Code должен знать: цифры, контакты, реквизиты, ссылки. Отличие от README.md: README — навигация, FACTS — данные.

### Шаблон FACTS.md

```markdown
# FACTS: Название проекта

Last Updated: YYYY-MM-DD

## Ключевые метрики
- Средний чек: 2 850 руб. [CONFIRMED: отчёт WB за январь 2026]
- Конверсия карточки: 4.2% [CONFIRMED: аналитика OZON Q4 2025]
- Комиссия площадки: 15% [CONFIRMED: договор от 01.01.2026]

## Контакты
- Менеджер WB: Иван Иванов, ivan@example.com
- Логистика: ООО "Доставка", +7-999-123-45-67

## Реквизиты
- ИП Иванов И.И., ИНН 123456789012
- Р/с: 40802810...
```

Создавай FACTS.md когда в проекте есть важные факты, которые нужно переиспользовать.

## 7. Кросс-референсы и теги

Главный принцип: **не дублируй — ссылайся.** Каждый факт описан в одном месте, остальные ссылаются на него.

### Теги

| Тег | Значение | Пример |
|-----|----------|--------|
| `[CANONICAL]` | Первоисточник, авторитетное определение | `Средний чек: 2 850 руб. [CANONICAL]` |
| `[REF: path#section]` | Ссылка на первоисточник | `[REF: MARKETPLACE_PRJ/FACTS.md#ключевые-метрики]` |
| `[CONFIRMED: source]` | Проверенная информация | `[CONFIRMED: отчёт WB за январь 2026]` |
| `[PLACEHOLDER: owner]` | Нужно заполнить | `[PLACEHOLDER: аналитик — уточнить ROI]` |

### Как это работает

1. В `MARKETPLACE_PRJ/FACTS.md` пишешь: `Комиссия WB: 15% [CANONICAL]`
2. В любом другом файле: `Комиссия WB: 15% [REF: MARKETPLACE_PRJ/FACTS.md#реквизиты]`
3. Если факт изменился — обновляешь только FACTS.md, а Claude Code найдёт все [REF:] и предложит обновить

## 8. Скиллы (Skills)

Скилл — это переиспользуемая команда для Claude Code. Вызывается через `/skill-name` в чате. Хранится в `.claude/skills/<skill-name>/SKILL.md`.

### Структура скилла

```
.claude/skills/ecom-manager/
├── SKILL.md          # Главный файл — инструкция для Claude Code
└── frameworks.md     # Дополнительные материалы (опционально)
```

### Формат SKILL.md

```markdown
---
name: ecom-manager
description: Анализ e-commerce метрик, ревью карточек, конкурентный анализ.
argument-hint: [задача или тема анализа]
---

# E-Commerce Manager

Ты — эксперт по e-commerce. Твоя задача — ...

## Компетенции
...

## Процесс работы
...

## Как использовать
- `/ecom-manager проанализируй карточку [URL]`
- `/ecom-manager юнит-экономика для [товар]`
```

---

### 8.1. Скилл: E-Commerce Manager

**Файл:** `.claude/skills/ecom-manager/SKILL.md`

**Назначение:** аналитика e-commerce, ревью карточек, расчёт юнит-экономики, конкурентный анализ.

**Что должен уметь:**

- Анализ карточки товара (заголовок, фото, описание, отзывы, позиционирование)
- Расчёт юнит-экономики: себестоимость → комиссия → логистика → маржа
- Конкурентный анализ: сравнение цен, позиций, рейтингов
- Анализ воронки: трафик → показы → клики → корзина → заказ → выкуп
- Рекомендации по оптимизации на основе данных

**Встроенные фреймворки:**

- Unit-экономика (формулы расчёта маржинальности с учётом комиссий площадок)
- JTBD (Jobs to be Done) — для анализа потребностей покупателя
- Метрики-пирамида: GMV → orders → AOV → conversion → traffic

**Команды:**
- `/ecom-manager анализ карточки [описание/ссылка]`
- `/ecom-manager юнит-экономика [параметры товара]`
- `/ecom-manager конкуренты [категория]`
- `/ecom-manager воронка [данные]`

**Интеграция с ролью:** при вызове скилл подгружает знания из `_PROMPTS/_ROLES/Head_of_E-Commerce.md`.

---

### 8.2. Скилл: Project Manager

**Файл:** `.claude/skills/project-manager/SKILL.md`

**Назначение:** управление бэклогом, планирование спринтов, статус-отчёты, приоритизация.

**Что должен уметь:**

- Читать и обновлять `backlog/backlog.xlsx`
- Приоритизировать задачи (по срочности/важности, матрица Эйзенхауэра)
- Формировать спринт: отобрать задачи из бэклога на неделю/2 недели
- Генерировать статус-отчёт: что сделано, что в работе, блокеры
- Декомпозиция крупных задач на подзадачи

**Команды:**
- `/project-manager статус` — текущий статус всех проектов из бэклога
- `/project-manager спринт [дата]` — сформировать спринт
- `/project-manager приоритизация` — ревью и пересортировка приоритетов
- `/project-manager декомпозиция [задача]` — разбить задачу на шаги

**Правила работы с бэклогом:**
- Единый файл `backlog/backlog.xlsx` — Single Source of Truth
- Не менять структуру колонок без явной команды
- Новые задачи — только добавлением строк (не перенумеровывать)
- Статусы: `TO DO`, `WIP`, `ON PRIORITY`, `DONE`, `ON HOLD`
- Приоритеты 1–100, без дубликатов, абсолютные

---

### 8.3. Скилл: Работа с Excel

**Файл:** `.claude/skills/excel-worker/SKILL.md`

**Назначение:** чтение, анализ и модификация Excel-файлов. Основной инструмент для работы с выгрузками маркетплейсов, финансовыми моделями, отчётами.

**Что должен уметь:**

- Читать и анализировать XLSX (сводные таблицы, фильтры, поиск аномалий)
- Модифицировать данные: обновить ячейки, добавить строки, пересчитать формулы
- Конвертировать: XLSX → Markdown, XLSX → CSV
- Строить сводки: суммы, средние, группировки по категориям

**Жёсткие правила:**

| Правило | Почему |
|---------|--------|
| Всегда `openpyxl.load_workbook()` | `Workbook()` создаёт новый файл и убивает форматирование |
| Менять только указанные ячейки | Чтобы не сломать остальные данные |
| Сохранять как `filename_vX.Y.xlsx` | Версионирование: X — крупные изменения, Y — мелкие |
| Даты — в формате Excel date | Не строкой, а настоящей датой |
| Заголовки — жирным | При создании новых таблиц |

**Команды:**
- `/excel-worker анализ [файл.xlsx]` — прочитать и дать сводку
- `/excel-worker обнови [файл.xlsx] [что изменить]` — модификация
- `/excel-worker конвертируй [файл.xlsx]` — в Markdown или CSV
- `/excel-worker сводка [файл.xlsx] по [колонка]` — группировка и агрегация

## 9. Роли (_PROMPTS/_ROLES/)

Роль — глубокий экспертный контекст. Если скилл — это «что делать», то роль — «кто отвечает».

Claude Code загружает роль, когда тема запроса совпадает с её специализацией. Роли хранятся в `_PROMPTS/_ROLES/`.

### Обязательные роли для e-com менеджера

**1. Head_of_E-Commerce.md** — главная роль. Включает:
- Компетенции: маркетплейсы, D2C, unit-экономика, JTBD, A/B-тесты
- Методологию анализа: ориентация → данные → JTBD → узкое место → стратегия → план → метрики
- Реальный опыт и кейсы

**2. Project_Manager.md** — для задач планирования и управления:
- Приоритизация, декомпозиция, спринты
- Работа с бэклогом
- Статус-отчёты и коммуникация

**3. Analyst.md** (опционально) — для глубокой аналитики:
- Работа с данными, SQL, Python
- Визуализация, дашборды
- A/B-тесты, когортный анализ

### Шаблон роли

```markdown
# Роль: Название

## Кто ты
Опытный [специализация] с [X лет] опытом в [область].

## Компетенции
1. ...
2. ...
3. ...

## Методология
Когда к тебе приходят с задачей, ты:
1. Уточняешь контекст и цель
2. Анализируешь данные
3. Даёшь рекомендации с обоснованием

## Принципы
- Решения на основе данных, не мнений
- Сначала узкое место, потом оптимизация
- Простое решение > сложное

## Чего не делаешь
- Не даёшь советов без данных
- Не соглашаешься ради вежливости
```

### Правила выбора роли

В CLAUDE.md укажи:
```markdown
## Role Selection
- E-commerce вопросы → _PROMPTS/_ROLES/Head_of_E-Commerce.md
- Планирование и задачи → _PROMPTS/_ROLES/Project_Manager.md
- Аналитика данных → _PROMPTS/_ROLES/Analyst.md
```

Claude Code сам выберет подходящую роль по контексту запроса.

## 10. Скрипты и автоматизация

Скрипты — это готовые инструменты, которые Claude Code может запускать. Менеджеру нужны три вещи: бэкап, конвертация, отчёты.

### Обязательные скрипты

**1. backup_to_git.zsh** — бэкап на GitHub

```bash
#!/bin/zsh
# Использование: ./scripts/backup_to_git.zsh "комментарий"
COMMENT=${1:-"backup: $(date +%Y-%m-%d_%H-%M-%S)"}
git add -A
git commit -m "$COMMENT"
git push origin main
```

**2. convert_xlsx_to_md.py** — Excel → Markdown (для читаемости в Git)

```bash
uv run --with openpyxl scripts/convert_xlsx_to_md.py файл.xlsx
```

**3. md_to_html.py** — Markdown → HTML/PDF (для отчётов и презентаций)

```bash
uv run --with markdown scripts/md_to_html.py отчёт.md -o отчёт.html
```

### Алиасы для .zshrc

```bash
# Добавь в ~/.zshrc:
alias backup="./scripts/backup_to_git.zsh"
alias xlsx2md="uv run --with openpyxl scripts/convert_xlsx_to_md.py"
alias md2html="uv run --with markdown scripts/md_to_html.py"
```

### README.md в scripts/

Обязательно создай `scripts/README.md` с описанием каждого скрипта: что делает, как запускать, какие зависимости.

## 11. Бэклог задач

Единый файл `backlog/backlog.xlsx` — Single Source of Truth для всех задач.

### Структура колонок

| Колонка | Описание |
|---------|----------|
| n | Уникальный ID задачи (целое число, никогда не переиспользовать) |
| PRJ | Код проекта (MARKETPLACE, DTC, ANALYTICS) |
| Название задачи | Короткий action-oriented заголовок, ~80 символов |
| Описание задачи | Что сделать, критерии приёмки |
| Приоритет | 1–100 (1 = самый важный). Абсолютный, без дубликатов |
| Дедлайн | YYYY-MM-DD (Excel date). Пусто, если нет дедлайна |
| Спринт | Дата окончания спринта YYYY-MM-DD |
| Статус | `TO DO` / `WIP` / `ON PRIORITY` / `DONE` / `ON HOLD` |
| Комментарий | Последний актуальный апдейт (перезаписывать) |

### Workflow

1. Новые задачи пишешь в `backlog/new-tasks.md` (свободный формат)
2. Claude Code или скрипт переносит их в `backlog.xlsx` с присвоением ID
3. Приоритизируешь вручную или через `/project-manager приоритизация`
4. Работаешь по спринтам, обновляя статусы

## 12. Git и GitHub

### 12.1. Первичная настройка

```bash
# 1. Инициализация
git init
git lfs install

# 2. Настройка LFS для тяжёлых файлов
git lfs track "*.xlsx"
git lfs track "*.pdf"
git lfs track "*.mp3"
git lfs track "*.mp4"
git lfs track "*.png"
git lfs track "*.jpg"

# 3. Создание .gitignore (см. шаблон ниже)

# 4. Первый коммит
git add -A
git commit -m "init: создание репозитория"

# 5. Подключение GitHub (создай приватный репозиторий на github.com)
git remote add origin git@github.com:USERNAME/REPO.git
git push -u origin main
```

### 12.2. Ежедневный бэкап

Минимум раз в день:

```bash
backup "описание что изменилось"
# или полная команда:
./scripts/backup_to_git.zsh "добавлены отчёты за январь"
```

### 12.3. Командная работа

**Модель для менеджерского репо (упрощённая):**

- `main` — стабильная версия, всегда рабочая
- `feature/название-задачи` — ветка под конкретную задачу

**Как поделиться с коллегой:**

1. Добавь коллегу как collaborator на GitHub (Settings → Collaborators)
2. Коллега клонирует: `git clone git@github.com:USERNAME/REPO.git`
3. Коллега получает весь контекст: CLAUDE.md, скиллы, роли, бэклог
4. Каждый работает в своей ветке, мерджит через Pull Request

**Pull Request даже для одного человека** — полезен как точка ревью: Claude Code может проверить изменения перед вливанием в main.

### 12.4. Git LFS

Excel, PDF, изображения, видео — бинарные файлы, которые Git хранит неэффективно. Git LFS решает эту проблему.

Что трекать через LFS:
```
*.xlsx, *.xls, *.pdf, *.png, *.jpg, *.jpeg, *.gif
*.mp3, *.wav, *.mp4, *.mov, *.avi, *.mkv
*.zip, *.rar, *.7z
```

Всё настраивается один раз через `git lfs track`, записывается в `.gitattributes`.

## 13. Память Claude Code

### 13.1. Auto Memory

Claude Code автоматически ведёт файл `.claude/memory/MEMORY.md` — запоминает паттерны, предпочтения, решения между сессиями.

**Что сохраняется:** стабильные паттерны, архитектурные решения, предпочтения пользователя, решения повторяющихся проблем.

**Что НЕ сохраняется:** текущие задачи, незавершённая работа, непроверенные выводы.

**Лимит:** 200 строк в MEMORY.md. Для деталей — отдельные файлы (например, `debugging.md`, `patterns.md`) со ссылками из MEMORY.md.

### 13.2. Правила (.claude/rules/)

Файлы в `.claude/rules/` загружаются автоматически в каждой сессии. Используй для правил, которые важны всегда, но не нужны в CLAUDE.md.

**Пример: `.claude/rules/error-learning.md`**

```markdown
# Error Learning

После бага, упавшего скрипта или 2+ неудачных попыток:

1. Запиши в `.claude/data/error-log.md`:
   - Симптом (что случилось)
   - Причина (почему)
   - Исправление (что сделано)
   - Предотвращение (как не повторить)

2. Обнови инструкции, если ошибка выявила пробел в правилах.

3. Правило трёх: кодифицируй правило сразу, только если фикс очевиден.
   Иначе жди 3 повторения одного класса ошибки.
```

### 13.3. Распределённая память

Помимо Auto Memory, информация хранится распределённо:

| Что | Где |
|-----|-----|
| Навигация по папке | README.md в каждой папке |
| Ключевые факты | FACTS.md в проектных папках |
| Задачи | backlog/backlog.xlsx |
| Ошибки и уроки | .claude/data/error-log.md |
| Общие правила | CLAUDE.md + .claude/rules/ |
| Экспертные знания | _PROMPTS/_ROLES/ |
| Версионирование | Git history |

## 15. Типичные ошибки

| Ошибка | Почему плохо | Как правильно |
|--------|-------------|---------------|
| Сложная структура с первого дня | Создаёшь 20 пустых папок, потом путаешься | Начни с 2–3 проектов, расти органически |
| Не обновляешь README.md | Claude Code не знает актуальную структуру | Обновляй при каждом изменении папки |
| Дублируешь информацию | Разные версии правды в разных файлах | Используй `[REF:]` теги |
| Коммитишь .env / секреты | Утечка ключей и паролей | Проверь .gitignore ДО первого коммита |
| Не делаешь бэкапы | Потеря работы | `backup "комментарий"` каждый день |
| Перегруженный CLAUDE.md | Claude Code путается в противоречивых правилах | До 150 строк, выноси детали в скиллы и роли |
| Всё в корне | Хаос из 50 файлов | Проекты в `*_PRJ/`, утилиты в `scripts/` |
| Excel через `Workbook()` | Убивает форматирование существующего файла | Только `load_workbook()` для существующих файлов |
